Os clientes não precisam se preocupar com aspectos relativos de loca1ização dos objetos que estiver usando, com os servidores que manipulam tais objetos. 0 servidor pode estar rodando na mesma máquina que seu cliente, numa máquina diferente na mesma rede local, ou mesmo em uma máquina situada a milhares de quilômetros de distância. Todavia, apesar de a maioria dos servidores rodar como processos de usuário, alguns servidores de tarefas de baixo nível, como os servidores de memória ou de processos, rodam como linhas de controle no kernel. Tal distinção também é invisível para os clientes. 0 protocolo de chamadas remotas a procedimentos que conversa com servidores implementados, como processos de usuários ou dentro do kernel, é exatamente o mesmo em todos os casos possíveis. Dessa forma, o cliente se preocupa tão-somente com o que deseja fazer, não com a localização dos objetos e dos servidores, embora, quando julgar conveniente, possa determinar o local onde os objetos serão armazenados. Além do mais, o sistema tem um conjunto razoável de defaults, de forma que, por exemplo, arquivos são muitas vezes armazenados no servidor de arquivos local, a menos que haja uma razão para se usar um servidor diferente. Não raro, o servidor local é o mais adequado.

14.2.1.Capacidades

Os objetos são identificados e protegidos de uma mesma maneira, usando tíquetes especiais denominados capacidades. Para criar um objeto, o cliente deve fazer uma chamada remota a procedimento com o servidor apropriado, especificando tudo o que ele deseja. 0 servidor cria então o objeto e retoma uma capacidade ao cliente. Nas operações subseqüentes, o cliente preciso apenas apresentar a capacidade para identificar o objeto.
