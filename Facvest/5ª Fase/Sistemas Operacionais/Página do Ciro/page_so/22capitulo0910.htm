Os itens de uma mensagem correspondem a identificação dos procedimentos e aos parâmetros. Uma mensagem com n parâmetros terá n+1 campos, um identifica o procedimento e os demais os parâmetros. De posse das informações sobre os tipos de parâmetros, torna-se possível realizar qualquer conversão que se torne necessária.
Para representar as informações nas mensagens, deve-se encontrar um padrão para as redes, ou forma canônica, e exigir que todos os servidores convertam suas representações internas para este formato enquanto fazem a ordenação dos parâmetros. Este método, às vezes, é ineficiente, pois o cliente precisa converter a mensagem, e o servidor necessita converter novamente quando a mensagem chegar, exigindo duas conversões, quando poderia não ser necessária nenhuma.
Pode-se usar outra abordagem, o cliente usa seu próprio formato nativo, e indica o formato usado no primeiro byte da mensagem. Na chegada da mensagem, o stub do servidor examina o primeiro byte para verificar o formato do cliente, se for o mesmo do servidor não há necessidade de conversão.
Em muitos sistemas baseados nas chamadas remotas a procedimento, a geração dos procedimentos stub é automática. Dada uma especificação do procedimento do servidor e as regras para codificação, o formato da mensagem é determinado, tornando possível fazer com que o compilador leia a especificação do servidor e gere um stub de cliente que empacote seus parâmetros. O compilador também pode produzir um stub de servidor que desempacote os parâmetros e chame o servidor.
O maior problema ligado às chamadas remotas a procedimentos é a passagem de ponteiros, pois estes só têm significado no espaço de endereçamento do processo no qual está sendo usado.
