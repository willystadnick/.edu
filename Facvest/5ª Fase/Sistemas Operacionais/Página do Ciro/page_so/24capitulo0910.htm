Perda de mensagens solicitando serviço
Basta fazer co que o kernel inicialize um temporizador quando do envio de uma mensagem de solicitação. Se este expirar antes da chegada de uma resposta ou de uma mensagem de reconhecimento, o kernel deve enviar a mensagem novamente. Pode ocorrer a perda de uma série de mensagens, fazendo com que o kernel desista das retransmissões e conclua, as vezes erroneamente, que o servidor saiu do ar.

Perda de mensagens com resposta
Uma solução seria utilizar novamente o recurso do temporizador, mas, neste caso, o kernel do cliente não está seguro a respeito do motivo pelo qual a resposta não chegou. Algumas operações podem ser repetidas de maneira segura, sem que essa repetição cause problemas, uma requisição com tal propriedade é chamada de idempotente. Uma forma de tentar resolver este problema é tentar estruturar todas as solicitações de forma idempotente. Um outro método é diferenciar as mensagens originais das retransmissões, mantendo um bit adicional no cabeçalho. Também se pode pedir ao kernel do cliente para numerar as mensagens enviadas, o kernel do servidor mantendo controle desta numeração poderá distinguir se a mensagem é original ou uma retransmissão.

Quedas do servidor
Existem três linhas distintas para tratar esta questão. Em uma delas o kernel do cliente espera até que o servidor dê um novo boot, ou que o kernel do cliente faça uma nova ligação com outro servidor, e, em ambos os casos, tente a operação novamente.
